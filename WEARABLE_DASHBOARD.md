# Wearable signals dashboard

The /today dashboard reads from a wearable + manual habit primitive,
narrative-led layout, and a tight end-of-day resolution loop. This doc
explains the three layers, the wearable-driven habit primitive, and how
to extend it with a new metric.

## Three-layer dashboard

```
┌──────────────────────────────────────────────────────┐
│ Headline (LLM)                                       │  Layer 1: narrative
│ "You slept 7h 12m. HRV is up 9% from your baseline.  │
│  Reformer is at 6pm. This is a good day."            │
├──────────────────────────────────────────────────────┤
│ TODAY'S SIGNALS · 4 of 6 connected                   │  Layer 2: tiles
│ ┌─────────┬─────────┬─────────┬─────────┐            │
│ │ Steps   │ Active  │ Sleep   │ HRV     │            │
│ │ 6,847   │ 22 min  │ 7h 12m  │ 64 ms   │            │
│ │ ▓▓▓░░░  │ ▓▓▓▓░░  │         │         │            │
│ │ 12 min  │ 8 min   │ +18 min │ +9% vs  │            │
│ │ walk to │ left    │ vs avg  │ baseline│            │
│ │ goal    │         │         │ recover'│            │
│ └─────────┴─────────┴─────────┴─────────┘            │
│ THIS WEEK                                            │
│ ▂▄▆▅█▃▁  (sage when target hit, champagne otherwise) │
├──────────────────────────────────────────────────────┤
│ TODAY'S HABITS                                       │  Layer 3: actions
│ ● 10,000 steps   6,847 / 10,000 ▓▓▓░░  ← no checkbox │
│ ○ LED mask                              ← checkbox   │
│ ○ Vibration plate                       ← checkbox   │
└──────────────────────────────────────────────────────┘
```

Layer 1 — `lib/dashboard/headline.ts`. Cached per (userId, date) in the
`TodayHeadline` table. Pre-generated by an Inngest job in the user's
06:00–07:30 local window so /today doesn't pay the LLM cost on first
load. If the cache misses, the page falls back to a deterministic line
within a 4 s budget.

Layer 2 — `components/dashboard/SignalsSection.tsx`. Six tiles: steps,
active min, sleep, HRV, resting HR, energy. Tiles have three variants
(`value`, `progress`, `empty`). Empty tiles are visible-on-purpose
"Connect →" ghosts so absence is felt rather than hidden.

Layer 3 — habits live in `app/(app)/today/TodayView.tsx`. Wearable
habits don't show a checkbox; they show a sage check (resolved DONE),
an outline dot (still tracking), or a progress bar with `current /
target` and the pace caption.

## Wearable-driven habit primitive

```prisma
enum HabitTrackingMode { MANUAL  WEARABLE_AUTO  HYBRID }
enum MetricComparison  { GTE  LTE  EQ }

model Habit {
  // … existing fields
  trackingMode      HabitTrackingMode @default(MANUAL)
  metricKey         String?           // HealthDaily.metric, NOT a column
  metricTarget      Float?
  metricComparison  MetricComparison?
}
```

`metricKey` is the string used in the canonical row-per-metric
HealthDaily table — `"steps"`, `"sleepHours"`, `"activeMinutes"`,
`"restingHr"`, `"hrvMs"`. It is **not** a column name. Resolution looks
up by the `(userId, date, metric)` unique.

### Resolution shape

`lib/habits/wearable-resolution.ts`:

- `resolveWearableHabit(habitId, date)` — reads HealthDaily for that
  (date, metric) and returns DONE / MISSED / unresolved.
- `currentWearableProgress(habitId)` — live progress for today + an
  `onPace` estimate. `onPace = (value / target) >= (localHour / 17)`
  for GTE habits. LTE / EQ return `onPace: null`.

### Tracking modes

| Mode | What user sees | When it resolves |
| --- | --- | --- |
| `MANUAL` | Checkbox, edit-mode delete | On user tap |
| `WEARABLE_AUTO` | Progress bar, no tap | End-of-day (23:55–00:30 user-local) |
| `HYBRID` | Tap or auto | User tap wins; otherwise end-of-day fallback |

### Built-in presets

`lib/plans/presets.ts` — every preset that contains a step or sleep
habit (`lean_strong`, `wedding_countdown`, `strength_foundation`,
`marathon_ready`, `healthy_reset`, `glutes_curves`) ships with the
metric trio set, so a user who picks a preset gets wearable tracking
without thinking about it.

### Free-text inference

When the chat creates a habit via `create_full_plan` and the caller
omits `trackingMode`, `vita-tools.ts` runs `inferWearableFromTitle`
which catches `"10,000 steps"`, `"8 h sleep"`, `"30 active minutes"`,
etc. and fills the trio.

## End-of-day loop and late-day nudge

`lib/jobs/wearable.ts`:

### `resolveWearableHabits` (every 15 min)

Window: **23:55–00:30 user-local**. For every user with an active
HealthIntegration:

1. Find active habits with `trackingMode IN (WEARABLE_AUTO, HYBRID)`
2. Skip habits not due today per cadence (DAILY / WEEKDAYS /
   WEEKENDS / SPECIFIC_DAYS)
3. For HYBRID, skip if a manual completion already exists today
4. Call `resolveWearableHabit(habitId, today)`
5. Upsert HabitCompletion with `source: "WEARABLE_AUTO"`. DONE rows
   grant the habit's `pointsOnComplete`. Existing WEARABLE_AUTO rows
   that started as MISSED can be upgraded to DONE if the user crossed
   the line late (e.g. last walk before bed).

Idempotent — running twice produces the same row.

### `lateDayWearableNudge` (every 10 min)

Window: **22:00–23:59 user-local**. For every user:

1. Skip if no active HealthIntegration
2. Skip if `NotificationPreference.lateDayNudge` is OFF
3. Skip if a `lateDayNudge` push has already been logged today
   (1-per-user-per-day throttle)
4. Find a candidate GTE habit (steps / activeMinutes only — never HRV
   or sleep, those aren't actionable late at night)
5. Compute the gap. If `5% ≤ gap/target ≤ 25%`, send a push:
   - Steps: `"1,200 steps to 10,000. A 12-minute walk would do it."`
   - Active min: `"8 minutes left to your 30. A song's worth of brisk walking."`

The push goes through `lib/notifications/send.ts` with `essential: true`
to bypass quiet hours (the user explicitly opted in to a late-night
ping by enabling this category). The dedicated 1-per-day check makes
the global throttle redundant for this category.

### Morning recap

`buildHeadlineContext` in `lib/dashboard/headline.ts` includes
yesterday's `WEARABLE_AUTO`-sourced HabitCompletion rows. The prompt
sees lines like `Yesterday: 10,000 steps: hit, 30 active min: missed`
and can reference the outcome in the next morning's headline.

## Adding a new wearable-tracked metric

To wire up, e.g., a "150 g protein" wearable habit (assuming a HAE
nutrition feed lands in HealthDaily):

1. Confirm the metric exists in HealthDaily — `metric: "proteinG"` rows
   with sane units. The HAE bridge in `lib/health/process-hae.ts` is
   the place to add new metric writes.
2. Add the field name to `formatMetricValue` in `app/(app)/today/TodayView.tsx`
   so the habit row shows a friendly unit.
3. Optional: add a tile to `components/dashboard/SignalsSection.tsx`
   if it should appear on the dashboard.
4. Optional: add to the auto-actionable list in `lib/jobs/wearable.ts`
   (`metricKey: { in: ["steps", "activeMinutes", ...] }`) if a late-day
   nudge makes sense for it.
5. Update the chat-side decomposer: `inferWearableFromTitle` in
   `lib/vita-tools.ts` — add a regex for the title pattern.
6. Update `WEARABLE_DASHBOARD.md` (this doc).

That's it. No schema migration needed — the `Habit.metricKey` field is
a free string.

## Admin endpoints

| Endpoint | What it does |
| --- | --- |
| `GET /api/admin/bridge-hae-to-health` | Re-runs the daily roll-up so HaeDaily → HealthDaily for any backlogged HAE payloads. Idempotent. |
| `GET /api/admin/migrate-step-habits` | Dry-run: what would convert from MANUAL to WEARABLE_AUTO. Add `?confirm=1` to apply. |
| `GET /api/admin/test-wearable-loop?steps=8500` | Dry-run the late-day nudge + end-of-day resolution against your own user. Add `?write=1` to actually stage HealthDaily and create a test habit. Optional `?steps2=10500` to simulate end-of-day after more walking. |

All admin endpoints are gated to `ADMIN_EMAILS` (env var; defaults to
the project owner's address).
